# Codex CLI 实战案例教程

这份文档不再重复基础概念，而是直接用几个典型场景，演示你该如何向 Codex CLI 下指令、它通常会怎么推进、你应该看什么、最后怎么判断结果是否靠谱。

如果你是初学者，建议把这份文档和前两份一起使用：

- `codex_tutorial.md`：完整原理和使用方式
- `codex_cheatsheet.md`：日常复制即用的速查模板
- `codex_case_studies.md`：完整实战案例

---

## 1. 使用案例前，先记住一个原则

同一个任务，问法不同，结果质量会差很多。

下面这三种说法里，第三种通常最好。

### 差的问法

```text
帮我修一下这个 bug
```

问题：没有现象、没有范围、没有限制、没有验证要求。

### 一般的问法

```text
帮我修复登录失败的问题。
```

问题：知道目标，但仍然缺少边界和交付标准。

### 好的问法

```text
请帮我定位并修复登录失败的问题。

要求：
- 先梳理调用链和根因
- 优先做最小修复
- 不要改无关模块
- 改完后运行相关测试
- 最后告诉我根因、修改点和验证结果
```

问题提得越清楚，Codex CLI 越容易做出靠谱结果。

## 2. 案例一：第一次进入陌生仓库

这是最常见的场景。你刚进一个项目，不知道：

- 项目是做什么的
- 怎么启动
- 从哪开始看
- 哪些目录最重要

### 推荐提问

```text
请先阅读这个仓库，然后告诉我：
1. 这个项目是做什么的
2. 启动方式是什么
3. 主要目录分别负责什么
4. 我应该按什么顺序阅读代码
```

### 这个提问为什么好

因为它把“看看项目”拆成了 4 个明确输出。

这样 Codex 通常会：

- 查看目录结构
- 找 README、package.json、pyproject.toml、Makefile、Dockerfile 等入口文件
- 查启动命令
- 找应用入口
- 总结阅读顺序

### 你应该期待它给出的结果

一份有结构的结论，例如：

1. 项目用途
2. 启动命令
3. 关键目录说明
4. 推荐阅读路径

### 你应该检查什么

- 它说的启动命令是否真实存在
- 它提到的目录职责是否和代码结构一致
- 它推荐的入口文件是否合理

### 不够好的输出长什么样

比如它只说：

```text
这是一个 Web 项目，结构比较清晰，你可以从主入口开始看。
```

这种回答问题在于信息密度太低，基本没帮助。

### 进一步追问

如果你想继续深入，可以接着问：

```text
现在请重点解释用户认证流程，从路由入口一路追到核心 service。
```

## 3. 案例二：定位并修复一个明确 bug

假设你遇到的问题是：

“用户点击登录后返回 500”

### 差的问法

```text
登录有问题，帮我修一下。
```

### 推荐提问

```text
请帮我定位并修复“用户点击登录后返回 500”的问题。

要求：
- 先梳理请求入口、调用链和最可能的根因
- 优先做最小修复
- 不要改无关模块
- 改完后运行相关测试
- 最后告诉我根因、修改点和验证结果
```

### Codex 正常的推进方式

一个靠谱的流程通常是：

1. 找登录接口入口
2. 找控制器、服务层、数据库调用
3. 看错误处理和参数校验
4. 定位具体异常点
5. 实施最小修复
6. 跑相关测试或最小验证

### 你为什么要强调“最小修复”

因为如果不写这个限制，它可能会：

- 顺手重构认证模块
- 改动公共校验逻辑
- 改一堆你暂时不关心的文件

这在真实项目里通常不是你想要的。

### 你应该要求它输出什么

建议固定成这 5 项：

```text
最后请按以下格式输出：
1. 根因
2. 修改点
3. 为什么这样改
4. 验证结果
5. 剩余风险
```

### 你应该检查什么

- 根因是否具体，而不是空泛地说“某处逻辑异常”
- 修改是否只覆盖问题点
- 是否说明为什么不会破坏其他登录流程
- 是否真的做了验证

### 一个高质量输出的大致样子

不是只说：

```text
已修复登录问题。
```

而是应当类似：

- 根因：登录请求里缺少空值保护，导致服务层解构失败
- 修改点：在入口处增加参数校验，并统一返回 400
- 验证结果：相关测试通过，手动模拟缺失参数时不再报 500

这类结果才有工程价值。

## 4. 案例三：补单元测试

很多人会让 Codex “写测试”，但如果不加限制，它可能写出和项目风格不一致的测试。

### 推荐提问

```text
为 `src/utils/date.ts` 补充单元测试。

要求：
- 沿用当前测试框架和命名风格
- 先阅读现有测试写法
- 覆盖正常路径、边界情况和异常输入
- 不要修改生产代码，除非测试暴露了真实 bug
- 写完后运行相关测试
```

### 为什么这个问法更好

因为它限制了几个关键点：

- 测试风格要一致
- 要先参考现有测试
- 覆盖范围要完整
- 不能为了写测试而乱改生产代码

### Codex 这时通常会做什么

- 搜索现有测试目录
- 找同模块或同风格测试
- 打开目标文件
- 设计用例
- 写测试
- 运行相关测试

### 你要特别警惕什么

有些低质量结果会出现两种问题：

1. 测试只覆盖 happy path
2. 为了让测试好写，顺手改了一堆生产代码

如果你看到这种情况，就应该收紧指令：

```text
只补测试，不要改生产代码；如果发现真实 bug，先说明再修改。
```

### 什么算好测试

通常至少应覆盖：

- 正常输入
- 空输入
- 边界值
- 非法输入
- 返回值或副作用校验

## 5. 案例四：做代码审查

如果你让 Codex 做 review，默认应该让它站在“找问题”的角度，而不是“总结代码写得不错”。

### 推荐提问

```text
请 review 当前改动，重点关注：
1. 潜在 bug
2. 行为回归风险
3. 错误处理缺失
4. 测试覆盖不足

要求：
- 先列发现的问题，再给简短总结
- 如果没有发现明确问题，请直说，并说明残余风险
```

### 这个问法的关键价值

它会把输出重心放在“发现问题”而不是“给个总体评价”。

### 高质量 review 应该长什么样

理想输出通常是：

1. 问题 1：文件、位置、风险原因
2. 问题 2：文件、位置、风险原因
3. 问题 3：文件、位置、风险原因
4. 简短总结

### 低质量 review 的典型样子

```text
代码整体结构清晰，可读性不错，建议后续补充更多测试。
```

这种回答几乎没有价值。

### 你还可以进一步限制

比如：

```text
不要给泛泛而谈的代码风格建议，优先找真实 bug 和行为风险。
```

这会让结果更聚焦。

## 6. 案例五：新增一个小功能

假设你要在 CLI 里加一个 `--json` 选项。

### 推荐提问

```text
请在现有 CLI 中新增 `--json` 输出选项。

要求：
- 保持默认文本输出行为不变
- 沿用现有参数解析方式
- 不引入新依赖
- 更新相关测试
- 如有帮助信息或 README，请同步更新
```

### 为什么这条指令有效

因为你已经提前把几个关键风险堵住了：

- 不允许破坏默认行为
- 不允许引入额外依赖
- 不允许脱离现有实现模式
- 文档和测试要同步

### Codex 正确的处理路径

通常应该是：

1. 找 CLI 入口
2. 找参数解析逻辑
3. 找输出逻辑
4. 查现有测试
5. 实现 `--json`
6. 补测试
7. 更新文档

### 你应该重点检查什么

- 默认输出是否保持兼容
- `--json` 是否只影响输出格式
- 错误分支是否也考虑了 JSON 输出
- 帮助文本是否同步更新

## 7. 案例六：生成项目文档

让 Codex 写文档时，最怕它脱离仓库实际情况，写成空泛模板。

### 推荐提问

```text
请根据当前仓库生成一份新成员入门文档，写入 `docs/onboarding.md`。

要求：
- 内容必须基于当前代码结构和配置
- 包含启动方式、目录结构、开发命令和常见注意事项
- 面向第一次接触这个项目的人
- 不要写脱离仓库实际情况的空话
```

### 为什么要写“必须基于当前仓库”

因为文档类任务最容易出现“看起来很完整，其实没落地”的问题。

### 你应该检查什么

- 文档里的命令是否真实存在
- 文件路径和目录结构是否对应仓库
- 说明是否面向新人，足够易懂

### 如果结果还不够实用

可以继续追问：

```text
请再补充一个“第一次启动最容易踩坑”的章节，结合当前配置文件来写。
```

## 8. 案例七：只允许它先分析，不许直接改

有些任务你不想让它直接动手，比如：

- 涉及线上逻辑
- 改动风险高
- 你想先确认方向

### 推荐提问

```text
先分析这个问题，不要修改代码。

我想知道：
1. 涉及哪些文件
2. 调用链是什么
3. 最可能的根因是什么
4. 如果要修，最小改动方案是什么
```

### 这个模式适合什么时候

- 你对仓库不熟
- 你不确定问题到底在哪
- 你想先评估改动风险

### 下一步怎么接

如果分析合理，再跟一句：

```text
现在按你刚才说的最小方案实施，并运行相关验证。
```

这是一种很稳的两段式协作方式。

## 9. 案例八：限制只改指定文件

有时你知道问题范围很小，就应该直接限制改动边界。

### 推荐提问

```text
请修复这个问题，但只允许修改以下文件：
- `src/api/order.ts`
- `src/components/Checkout.tsx`

要求：
- 不要改其他文件
- 做最小修复
- 如果发现根因在别处，先告诉我，不要自行扩散修改
```

### 为什么这很有用

这能有效防止：

- 任务蔓延
- 顺手重构
- 无关文件被改动

对高风险仓库尤其重要。

## 10. 案例九：测试跑不起来时该怎么让它处理

很多仓库不是一进来就能跑测试，可能缺：

- 依赖
- 环境变量
- 数据库
- 服务端口

这时你应该要求它把阻塞项说明白，而不是含糊跳过。

### 推荐提问

```text
改完后请运行相关测试。
如果测试无法运行，请明确说明：
1. 卡在哪一步
2. 缺少什么条件
3. 你已经做到哪一步验证
不要直接跳过说明。
```

### 为什么这很重要

因为“已修改但未验证”与“已修改且验证通过”在工程上完全不是一回事。

## 11. 案例十：怀疑它方向跑偏时如何纠正

有时你会发现它开始：

- 搜索不相关文件
- 准备大改结构
- 修改范围越来越大

这时不要客气，直接收窄。

### 推荐纠偏指令

```text
停止当前方向。
只聚焦在 `src/auth/login.ts` 和 `src/api/session.ts`。
目标是做最小修复，不要扩展到认证模块其他部分。
```

### 为什么要这样说

因为“你方向不对”太抽象，而“只看这两个文件、只做最小修复”非常具体。

Codex CLI 更容易执行具体约束。

## 12. 一套适合真实工作的协作套路

如果你平时想把 Codex CLI 真正用起来，建议采用下面这套流程。

### 第一步：先给清晰任务

写清楚：

- 目标
- 范围
- 约束
- 验证要求

### 第二步：先让它建立上下文

除非任务极小，否则通常应该允许它先阅读相关代码。

### 第三步：要求最小修改

这会显著减少不必要风险。

### 第四步：要求验证

至少要知道：

- 它跑了什么
- 没跑成什么
- 还有什么风险

### 第五步：检查它的结论是否具体

好结果通常能回答：

- 根因是什么
- 改了哪里
- 为什么这样改
- 怎么验证的

## 13. 初学者练习路径

你可以按下面的顺序实际练习。

### 练习 1：仓库导览

目标：让它带你快速读懂项目。

建议指令：

```text
先阅读仓库，然后告诉我项目用途、启动方式、关键目录和建议阅读顺序。
```

### 练习 2：解释一个文件

目标：看它是否能准确解释逻辑。

建议指令：

```text
解释 `src/auth/session.ts` 的核心逻辑，重点说明 token 刷新流程。
```

### 练习 3：改一个小地方

目标：熟悉它如何做最小修改。

建议指令：

```text
把首页标题文案改成 Dashboard，保持样式不变，不要改无关文件。
```

### 练习 4：补测试

目标：看它是否能沿用现有测试风格。

建议指令：

```text
为 `src/utils/date.ts` 补充测试，沿用现有测试风格，覆盖边界情况。
```

### 练习 5：修一个小 bug

目标：练习完整的“分析 -> 修改 -> 验证”链路。

建议指令：

```text
请定位并修复这个报错，优先最小修改，改完后运行相关验证，并解释根因。
```

## 14. 最后总结：实战时最重要的不是“会不会提问”，而是“会不会收边界”

很多初学者以为关键是把需求说得“高级”。

其实真正关键的是这几件事：

1. 目标要具体
2. 范围要收窄
3. 约束要提前写
4. 验证要明确要求
5. 风险操作要谨慎审批

如果你能做到这 5 点，Codex CLI 在真实开发中的稳定性会高很多。
