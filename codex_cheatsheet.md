# Codex CLI 速查手册

这份文档适合已经看过基础教程、接下来希望“拿来就用”的用户。你可以把它当成日常使用 Codex CLI 的提示词模板库和操作清单。

---

## 1. 最常用的 10 句指令

### 1. 先读仓库

```text
先阅读这个仓库，然后告诉我项目用途、启动方式、主要目录结构和推荐阅读顺序。
```

### 2. 先分析，不要动代码

```text
先分析这个问题的可能根因和相关文件，不要立刻修改代码。
```

### 3. 做最小修复

```text
请做最小修复，不要改无关文件，不要顺手重构。
```

### 4. 保持现有风格

```text
沿用当前项目的代码风格、目录结构和命名方式，不要引入新的架构层。
```

### 5. 不引入依赖

```text
不要引入新依赖，优先复用现有工具和组件。
```

### 6. 改完要验证

```text
改完后运行相关测试，并告诉我实际结果。
```

### 7. 解释根因

```text
除了修复，请解释根因，以及为什么这个改动是正确的。
```

### 8. 限定范围

```text
只关注 `src/auth` 和 `src/api/login.ts`，不要扩散到其他模块。
```

### 9. 做 review

```text
请 review 当前改动，重点找 bug、行为回归和缺失测试。
```

### 10. 生成文档

```text
根据当前项目生成一份开发文档，写入指定文件，并保持内容面向新人可读。
```

## 2. 最实用的提示词模板

## 2.1 通用任务模板

```text
请帮我处理这个任务。

目标：
[写清楚你要的结果]

上下文：
[补充背景、报错、模块、文件]

约束：
- 只做必要修改
- 保持现有代码风格
- 不改无关文件

验证要求：
[运行测试 / 不运行测试 / 做静态检查]

交付要求：
[修改代码 / 解释根因 / 更新文档]
```

## 2.2 Bug 修复模板

```text
请帮我定位并修复这个问题。

现象：
[写报错或错误行为]

范围：
[相关模块、接口、页面]

要求：
- 先分析根因
- 优先最小修复
- 不要改无关逻辑
- 改完后运行相关测试

输出：
- 根因
- 修改点
- 验证结果
```

## 2.3 新功能模板

```text
请在现有项目中实现这个功能。

需求：
[写功能目标]

限制：
- 保持现有架构
- 不引入新依赖
- 尽量复用现有模块
- 不改变已有默认行为

补充要求：
- 更新相关测试
- 如有必要更新文档
```

## 2.4 重构模板

```text
请重构这部分代码。

目标：
[例如减少重复、提高可读性]

约束：
- 不改变对外接口
- 不改变行为
- 不做跨模块大搬家
- 优先做小范围整理

完成后：
- 说明重构前后的区别
- 运行相关测试
```

## 2.5 文档生成模板

```text
请基于当前仓库生成文档。

目标文档：
[例如 onboarding / API 使用说明 / 部署指南]

要求：
- 内容面向初学者
- 结合当前代码结构
- 不要写脱离仓库实际情况的空话
- 写入指定文件
```

## 3. 高质量协作的关键短句

你可以反复复用下面这些约束语句。

- `先分析，不要改代码`
- `如果有不确定的地方，请先说明假设`
- `只做最小修改`
- `不要改无关文件`
- `不要引入新依赖`
- `不要为了过测试去修改错误的测试预期`
- `优先复用现有实现`
- `保持对外接口不变`
- `改完后运行相关测试`
- `把确定事实和推测分开写`
- `先给我调用链，再决定怎么改`
- `不要做顺手重构`
- `先阅读相关文件，再动手`

## 4. 常见任务的现成提问法

## 4.1 读懂仓库

```text
请先阅读这个仓库，然后告诉我：
1. 这个项目是做什么的
2. 如何启动
3. 主要目录分别负责什么
4. 我应该从哪里开始读
```

## 4.2 查启动失败

```text
请帮我定位项目启动失败的原因。先检查启动命令、配置文件和环境变量要求，先分析，不要立刻大改。
```

## 4.3 修测试

```text
运行相关测试并定位失败原因，优先修复真实问题，不要为了让测试变绿而篡改合理预期。
```

## 4.4 补单元测试

```text
为这个模块补充单元测试，沿用当前测试框架和命名风格，覆盖正常路径和边界情况。
```

## 4.5 查接口 bug

```text
请定位这个接口返回 500 的根因，先梳理调用链和输入校验逻辑，再决定如何修复。
```

## 4.6 做代码审查

```text
请 review 当前改动，重点关注：
1. 潜在 bug
2. 行为回归
3. 异常处理缺失
4. 测试覆盖不足
```

## 4.7 生成功能说明

```text
根据当前实现生成一份功能说明文档，包含入口、调用链、关键配置和注意事项。
```

## 5. 初学者最该避免的提问方式

下面这些说法通常不够好。

### 太空

```text
帮我看看
```

### 太大

```text
优化整个项目
```

### 太模糊

```text
这个地方不太对，改一下
```

### 没约束

```text
重构这个模块
```

更好的写法一般至少包含：

- 目标
- 范围
- 限制
- 验证要求

## 6. 什么时候要主动加限制

以下场景尤其建议加限制语句。

### 改线上问题时

加：

```text
优先最小修复，不要扩大改动面。
```

### 改老项目时

加：

```text
沿用现有写法，不要重新设计架构。
```

### 时间紧时

加：

```text
先给我最小可行修复方案。
```

### 担心误改时

加：

```text
只允许改这几个文件：[列出文件]
```

### 担心乱加依赖时

加：

```text
不要引入新依赖。
```

## 7. 让它输出更专业的结果

如果你不想只拿到“已经改好了”这种低信息量回复，可以要求固定输出结构。

例如：

```text
最后请按以下格式输出：
1. 根因
2. 修改点
3. 为什么这样改
4. 验证结果
5. 剩余风险
```

做 review 时也可以要求：

```text
先列发现的问题，再给简短总结。
```

---

## 8. 审批请求怎么看

当 Codex CLI 请求更高权限时，优先看下面几点。

### 命令是否和任务直接相关

比如修 bug 时请求运行测试，通常合理。  
如果只是解释代码却请求下载一堆依赖，就需要谨慎。

### 是否会改工作区外内容

如果会影响系统环境、全局目录或外部服务，需要更慎重。

### 是否具有破坏性

像删除、重置、覆盖这类动作要格外仔细看。

一条简单原则：

`看不懂就别批，先让它解释。`

## 9. 速用工作流

## 9.1 第一次进新仓库

顺序建议：

1. 让它解释仓库结构
2. 让它告诉你启动方式
3. 让它找主要入口文件
4. 让它解释一个核心模块

对应指令：

```text
先阅读仓库并告诉我项目用途、启动方式、关键目录和建议阅读顺序。
```

## 9.2 遇到一个 bug

顺序建议：

1. 先让它定位调用链
2. 再让它给出根因
3. 然后做最小修复
4. 最后运行验证

对应指令：

```text
请先定位这个 bug 的调用链和根因，再实施最小修复，并运行相关验证。
```

## 9.3 要加一个小功能

顺序建议：

1. 先找现有类似实现
2. 再要求沿用现有模式实现
3. 最后补测试和文档

对应指令：

```text
先找仓库里类似功能的实现方式，然后沿用相同模式完成这个功能，补充相关测试。
```

## 10. 适合新手的练习顺序

如果你刚开始用 Codex CLI，建议按这个顺序熟悉：

1. 解释目录结构
2. 解释某个文件
3. 修改一句文案
4. 修一个小 bug
5. 补一个测试
6. 生成一份文档
7. 做一次代码审查

先从低风险、易验证的任务开始，效果最好。

## 11. 一页版总结

记住下面这套最小原则就够用了：

### 提问时

- 目标写清楚
- 范围写清楚
- 限制写清楚
- 验证要求写清楚

### 协作时

- 先分析再决定是否修改
- 优先最小改动
- 尽量复用现有实现
- 改完要验证

### 审批时

- 看命令是否必要
- 看是否超出工作区
- 看是否有破坏性

### 判断结果时

- 是否基于真实代码
- 是否解释了根因
- 是否做了验证
