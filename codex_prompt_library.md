# Codex CLI 提示词库

这份文档专门整理适合 Codex CLI 的高质量提示词模板。目标不是追求“说得漂亮”，而是让你在真实项目里更稳定地得到可用结果。

你可以把它理解成一个按场景分类的“可直接复制使用”的提问库。

---

## 1. 使用说明

这些提示词不是必须逐字照抄，但建议保留它们背后的结构：

- 目标
- 范围
- 约束
- 验证要求
- 输出要求

如果你只说目标，不说边界，Codex CLI 更容易做出超范围修改。

如果你说了边界和验证要求，结果通常会明显更稳。

---

## 2. 通用模板

## 2.1 通用任务模板

```text
请帮我处理这个任务。

目标：
[写清楚想要的结果]

上下文：
[补充相关背景、报错、模块、文件]

范围：
[说明只关注哪些模块或文件]

约束：
- 只做必要修改
- 保持现有代码风格
- 不改无关文件

验证要求：
[运行测试 / 做静态检查 / 不需要验证]

输出要求：
[解释根因 / 列修改点 / 更新文档]
```

## 2.2 先分析后实施模板

```text
请先分析这个任务，不要直接修改代码。

我希望你先告诉我：
1. 涉及哪些文件
2. 调用链或依赖关系是什么
3. 最可能的根因或切入点是什么
4. 最小可行方案是什么
```

## 2.3 最小修改模板

```text
请做最小修改来解决这个问题。

要求：
- 不要改无关文件
- 不要顺手重构
- 保持现有行为和接口不变
- 如果发现需要扩大改动范围，先告诉我
```

## 3. 仓库理解类提示词

## 3.1 首次阅读仓库

```text
请先阅读这个仓库，然后告诉我：
1. 这个项目是做什么的
2. 启动方式是什么
3. 主要目录分别负责什么
4. 我应该从哪里开始阅读
```

## 3.2 找启动入口

```text
请帮我找出这个项目的启动入口、关键配置文件和本地开发命令。
```

## 3.3 梳理模块职责

```text
请解释这个项目中各个核心模块的职责，并指出它们之间的主要调用关系。
```

## 3.4 给出阅读顺序

```text
如果我是第一次接触这个仓库，请给我一个高效的阅读顺序，并说明每一步为什么这样安排。
```

---

## 4. 代码解释类提示词

## 4.1 解释单个文件

```text
请解释 `src/auth/session.ts` 的核心逻辑，重点说明 token 刷新流程和异常处理。
```

## 4.2 解释调用链

```text
请从这个路由入口开始，梳理完整调用链，直到核心 service 或数据库访问层。
```

## 4.3 解释业务流程

```text
请解释用户登录这条业务流程，从请求进入到返回响应为止，重点标出关键判断和错误分支。
```

## 4.4 区分事实和推测

```text
请把你能从代码里确定的事实和你的推测分开写。
```

## 5. Bug 排查与修复类提示词

## 5.1 定位 bug

```text
请帮我定位这个问题。

现象：
[写清楚错误行为]

要求：
- 先梳理调用链
- 找最可能的根因
- 先不要直接大改代码
```

## 5.2 定位并修复 bug

```text
请帮我定位并修复这个问题。

现象：
[写清楚错误行为或报错]

要求：
- 先分析根因
- 优先做最小修复
- 不要改无关模块
- 改完后运行相关测试
- 最后告诉我根因、修改点和验证结果
```

## 5.3 线上问题最小修复

```text
请针对这个线上问题给出最小修复方案。

要求：
- 优先降低风险
- 不做大规模重构
- 保持现有对外行为不变
- 如果有残余风险，请明确写出
```

## 5.4 限定文件范围修 bug

```text
请修复这个问题，但只允许修改以下文件：
- `src/api/order.ts`
- `src/components/Checkout.tsx`

如果你发现根因不在这些文件里，请先告诉我，不要自行扩大修改范围。
```

## 6. 测试相关提示词

## 6.1 运行并定位测试失败

```text
请运行相关测试并定位失败原因。

要求：
- 先找根因
- 优先修复真实问题
- 不要为了过测试去改错误的测试预期
```

## 6.2 补单元测试

```text
为 `src/utils/date.ts` 补充单元测试。

要求：
- 沿用当前测试框架和命名风格
- 先参考现有测试写法
- 覆盖正常路径、边界情况和异常输入
- 写完后运行相关测试
```

## 6.3 只补测试，不改生产代码

```text
请只补测试，不要修改生产代码；如果测试暴露了真实 bug，请先告诉我再修改。
```

## 6.4 说明测试阻塞项

```text
改完后请运行相关测试。
如果测试无法运行，请明确说明：
1. 卡在哪一步
2. 缺少什么条件
3. 你已经完成了哪些替代验证
```

## 7. 功能开发类提示词

## 7.1 新增一个小功能

```text
请在现有项目中实现这个功能。

需求：
[写功能目标]

要求：
- 保持现有架构和代码风格
- 优先复用现有实现
- 不引入新依赖
- 更新相关测试
```

## 7.2 保持默认行为不变

```text
请新增这个能力，但保持当前默认行为完全不变。
```

## 7.3 沿用已有实现模式

```text
先找仓库里类似功能的实现方式，然后沿用同样模式完成这次改动。
```

## 7.4 同步文档

```text
如果这次改动影响使用方式，请同步更新 README 或对应文档。
```

## 8. 重构类提示词

## 8.1 小范围重构

```text
请重构这部分代码。

目标：
[减少重复 / 提高可读性 / 降低嵌套复杂度]

约束：
- 不改变对外接口
- 不改变行为
- 不做跨模块大搬家
- 优先做小范围整理
```

## 8.2 清理重复逻辑

```text
请清理这个模块中的重复逻辑，但不要抽成跨模块公共库，先在模块内收敛。
```

## 8.3 重构后要求说明

```text
完成后请告诉我：
1. 重构前的问题
2. 重构后的结构
3. 为什么行为没有变化
4. 你做了什么验证
```

## 9. 代码审查类提示词

## 9.1 通用 review

```text
请 review 当前改动，重点关注：
1. 潜在 bug
2. 行为回归风险
3. 错误处理缺失
4. 测试覆盖不足
```

## 9.2 强调先列问题

```text
请先列发现的问题，再给简短总结。不要先做泛泛而谈的总体评价。
```

## 9.3 不要只提风格建议

```text
不要优先给代码风格建议，优先找真实 bug、逻辑漏洞和行为风险。
```

## 9.4 无问题时也要说明残余风险

```text
如果没有发现明确问题，请直接说明，并补充残余风险或测试缺口。
```

## 10. 文档生成类提示词

## 10.1 生成新成员入门文档

```text
请根据当前仓库生成一份新成员入门文档。

要求：
- 内容基于当前实际代码结构
- 包含启动方式、目录结构、开发命令和常见注意事项
- 面向第一次接触这个项目的人
- 写入指定文件
```

## 10.2 生成模块说明文档

```text
请基于当前实现生成这个模块的说明文档，包含入口、调用链、关键配置和常见坑。
```

## 10.3 生成部署说明

```text
请根据当前项目中的配置和脚本，生成一份部署说明，不要写与仓库实际情况无关的模板内容。
```

## 11. 输出格式控制类提示词

有时不是任务本身有问题，而是输出不够好读。这时可以直接限定输出格式。

## 11.1 修 bug 后固定输出格式

```text
最后请按以下格式输出：
1. 根因
2. 修改点
3. 为什么这样改
4. 验证结果
5. 剩余风险
```

## 11.2 分析任务时固定输出格式

```text
请按以下格式回复：
1. 涉及文件
2. 调用链
3. 根因判断
4. 最小修复方案
```

## 11.3 review 时固定输出格式

```text
请按“发现的问题 -> 风险原因 -> 建议处理”的格式输出。
```

这些短句非常有用，建议你在很多任务里都加上。

- `先分析，不要改代码`

- `不要改无关文件`
- `不要顺手重构`

- `保持对外接口不变`
- `优先复用现有实现`
- `如果需要扩大修改范围，先告诉我`
- `把确定事实和推测分开写`
- `改完后运行相关测试`
- `如果无法验证，请明确说明原因`

## 13. 纠偏类提示词

如果你感觉 Codex CLI 开始跑偏，可以直接用这些话收回来。

## 13.1 收窄范围

```text
停止当前方向，只聚焦在 `src/auth/login.ts` 和 `src/api/session.ts`。
```

## 13.2 禁止继续扩散

```text
不要继续扩展到其他模块，先把当前最小问题解决。
```

## 13.3 改成先分析

```text
先不要继续修改，先告诉我你现在判断的根因和接下来打算改哪里。
```

## 13.4 禁止大改

```text
我现在不需要重构，只需要一个低风险的最小修复。
```

## 14. 审批和验证类提示词

## 14.1 命令执行前先解释

```text
如果你需要申请更高权限，请先说明原因和这条命令的目的。
```

## 14.2 无法验证时必须说明

```text
如果你没法运行测试或验证，请不要省略，直接告诉我阻塞点。
```

## 14.3 区分“已修改”和“已验证”

```text
请明确区分：哪些内容只是改了，哪些内容已经验证通过。
```

## 15. 适合初学者直接复制的完整提示词

## 15.1 读仓库

```text
请先阅读这个仓库，然后告诉我：
1. 这个项目是做什么的
2. 如何启动
3. 主要目录负责什么
4. 我应该从哪里开始看代码
```

## 15.2 修 bug

```text
请帮我定位并修复这个问题。

现象：
[在这里写报错或错误行为]

要求：
- 先分析根因
- 优先最小修复
- 不要改无关文件
- 改完后运行相关测试

最后请告诉我：
1. 根因
2. 修改点
3. 验证结果
4. 剩余风险
```

## 15.3 补测试

```text
为这个模块补充测试。

要求：
- 沿用当前测试风格
- 覆盖正常路径和边界情况
- 不要修改生产代码，除非发现真实 bug
- 写完后运行相关测试
```

## 15.4 做 review

```text
请 review 当前改动，重点找：
1. 潜在 bug
2. 行为回归
3. 异常处理缺失
4. 测试覆盖不足

要求：
- 先列问题，再给简短总结

```

## 15.5 生成文档

```text
请根据当前仓库生成一份文档，写入指定文件。

要求：
- 内容基于仓库真实结构
- 面向初学者
- 包含必要命令、目录说明和注意事项
- 不要写空泛模板内容
```

## 16. 最后建议：把提示词拆成“目标 + 约束 + 验证”

如果你不想每次都翻整份文档，只记住这一条就够了。

大多数任务都可以用下面的最小骨架表达：

```text
目标：[我要什么]
约束：[不要做什么 / 只能改哪里 / 要保持什么不变]
验证：[改完怎么证明是对的]
```

例如：

```text
目标：修复登录失败
约束：最小修改，不改无关模块
验证：运行相关测试，并说明根因
```

这已经足够比大多数随手提问方式更有效。
