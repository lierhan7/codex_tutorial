# Codex CLI 初学者详细教程

本文面向第一次接触 Codex CLI 的用户，目标不是只告诉你“命令怎么敲”，而是让你理解它的工作方式、常见限制、正确协作姿势，以及如何把它真正用进日常开发。

---

## 1. Codex CLI 是什么

Codex CLI 可以理解为“运行在终端里的 AI 编程助手”。

它和普通聊天机器人最大的区别在于：

- 它可以直接查看你当前工作目录里的代码
- 它可以运行终端命令
- 它可以修改文件
- 它可以根据权限设置申请更高权限
- 它会在真实代码库里完成任务，而不是只给你一段示例代码

你可以把它当成一个会读代码、会改代码、会跑命令、会解释问题的协作型工程师。

---

## 2. 适合用 Codex CLI 做什么

Codex CLI 特别适合以下场景：

- 快速熟悉一个陌生仓库
- 修复小到中等规模 bug
- 批量重构代码
- 补测试
- 写文档
- 查找报错原因
- 实现一个明确的小功能
- 做代码审查
- 解释某个模块的逻辑

不太适合直接一句话丢给它的大而空任务：

- “把整个项目重写一遍”
- “帮我做个完美架构”
- “把这个仓库全部优化”

更好的提问方式是把任务缩小、写具体、给边界。

---

## 3. 一个最重要的认知：它在你的工作区里做事

Codex CLI 不是只输出文本，它通常会在当前目录执行真实操作。

这意味着你需要知道三件事：

1. 它读的是你当前目录下的文件
2. 它改的是真实文件，不是虚拟副本
3. 它会受到沙箱、权限、网络和工具可用性的约束

所以在使用前，最好先进入正确目录：

```bash
cd /path/to/your/project
codex
```

如果你在错误目录启动，Codex 看到的就不是你想让它处理的项目。

---

## 4. 基本工作流程

一个典型的 Codex CLI 使用流程通常是：

1. 进入项目目录
2. 启动 Codex CLI
3. 用自然语言描述任务
4. Codex 先阅读代码和目录结构
5. Codex 提出执行动作，或直接开始修改
6. 它运行命令、编辑文件、执行测试
7. 最后给你结果总结，包括改了什么、验证了什么、还有什么风险

你可以把它理解为下面这种协作模式：

- 你负责定义目标和边界
- Codex 负责探索、实现、验证和说明

## 5. 第一次应该怎么提问

### 不推荐

```text
帮我看看这个项目
```

问题是范围太大，没有重点。

### 更推荐

```text
请先阅读项目结构，然后告诉我这个仓库的启动方式、主要模块和开发入口文件。
```

或者：

```text
帮我定位登录失败的问题，先不要修改代码，先告诉我可能原因和相关文件。
```

或者：

```text
为 users 接口补充单元测试，沿用项目现有测试风格，改完后运行相关测试。
```

好的请求通常包含这些信息：

- 目标是什么
- 是否允许直接改代码
- 是否需要先分析再动手
- 是否要运行测试
- 是否要遵循现有风格

## 6. 常见交互方式

你可以把指令分成 4 类。

### 6.1 让它解释

例如：

```text
解释一下 src/auth/session.ts 的核心逻辑，重点看 token 刷新流程。
```

适合你看不懂一段代码时使用。

### 6.2 让它修改

例如：

```text
把首页标题从 Welcome 改成 Dashboard，并保持现有样式不变。
```

适合明确的小改动。

### 6.3 让它排查

例如：

```text
运行测试并定位失败原因，先不要大改结构，优先做最小修复。
```

适合 bug 修复和 CI 失败排查。

### 6.4 让它生成内容

例如：

```text
根据当前项目结构生成一份开发者入门文档，写入 docs/getting-started.md。
```

适合写文档、脚本、配置说明、迁移指南。

## 7. Codex CLI 常见行为，你需要提前知道

### 7.1 它通常会先阅读代码

这不是“变慢”，而是正常行为。

高质量修改依赖上下文。一个靠谱的 Codex 通常会先：

- 列目录
- 搜索关键字
- 打开相关文件
- 看测试和配置

如果它没看上下文就直接改，大概率质量一般。

### 7.2 它会给你中间进度

比如它会告诉你：

- 正在查看项目结构
- 正在定位某个模块
- 准备修改哪些文件
- 正在运行测试

这类信息的作用是让你知道它没“卡死”，而是在推进任务。

### 7.3 它通常会做最小必要修改

成熟的工作方式不是“推翻重来”，而是：

- 先理解现有模式
- 尽量复用已有结构
- 只改和任务相关的部分

如果你希望大改，最好明确说明。

## 8. 权限、沙箱和审批

这是很多初学者最容易忽略的部分。

Codex CLI 一般运行在受限环境里，常见限制包括：

- 只能读写当前工作区
- 网络访问受限
- 某些目录不能写
- 某些命令需要你的批准

### 8.1 什么情况下它会申请更高权限

比如：

- 安装依赖
- 访问网络
- 写入工作区外的目录
- 运行可能影响系统环境的命令

这时它通常不会偷偷执行，而是发起批准请求。

### 8.2 你应该怎么判断是否批准

看三件事：

1. 这个命令是否与你当前任务直接相关
2. 它会不会改动系统或仓库外内容
3. 风险是否可接受

如果任务只是“解释代码”，却请求执行下载依赖，那就值得警惕。

如果任务是“安装并运行测试”，那么安装依赖通常是合理请求。

### 8.3 初学者建议

- 看清命令用途再批准
- 不理解就先拒绝，再让它解释原因
- 对删除、重置、覆盖类操作格外谨慎

## 9. 它如何修改文件

Codex CLI 在好的工作流里，通常不会粗暴整文件覆盖，而是做有针对性的修改。

理想状态下，它会：

- 先读文件
- 理解上下文
- 只改必要部分
- 保持原有风格

你可以主动提出约束，例如：

```text
只做最小修改，不要调整无关代码风格。
```

或者：

```text
保持现有命名方式和目录结构，不要引入新的依赖。
```

这类约束非常有用。

---

## 10. 如何提需求，效果会明显更好

下面是一个很实用的模板：

```text
目标：
约束：
涉及文件或模块：
是否允许改代码：
是否需要运行测试：
交付结果：
```

例如：

```text
目标：修复用户注册接口返回 500 的问题
约束：优先最小改动，不要改数据库结构
涉及文件或模块：auth、users、api/register
是否允许改代码：允许
是否需要运行测试：需要，至少运行相关测试
交付结果：修改代码并说明根因
```

这个格式的好处是，Codex 会少走很多弯路。

## 11. 常见高质量指令示例

### 示例 1：先分析再改

```text
先阅读项目结构并定位支付回调逻辑。先告诉我调用链和潜在问题，不要立刻修改。
```

### 示例 2：做最小修复

```text
修复这个测试失败，尽量只改生产代码里真正有问题的部分，避免为了过测试而改测试预期。
```

### 示例 3：遵循现有风格

```text
为订单模块新增导出 CSV 功能，沿用现有 service/controller 分层，不要重新设计架构。
```

### 示例 4：让它做代码审查

```text
请 review 当前改动，重点找 bug、行为回归风险和缺失测试，不要只给概述。
```

### 示例 5：生成文档

```text
根据当前仓库生成一份新成员入门文档，包含启动方式、目录结构、开发命令和常见坑，写入 docs/onboarding.md。
```

## 12. 如何让它做“正确的事”而不是“看起来很努力”

初学者常见误区是只给目标，不给边界。

比如一句：

```text
优化这个项目
```

这会导致结果很虚。因为“优化”可能意味着：

- 提升性能
- 改善可读性
- 减少重复代码
- 补测试
- 调整目录

更好的说法是：

```text
请检查首页首屏渲染慢的问题，优先找最可能的性能瓶颈，并给出最小可行修复。
```

或者：

```text
请清理 users 模块中的重复校验逻辑，但不要跨模块抽象公共库。
```

范围越明确，输出越稳。

## 13. 你应该经常使用的限制语句

以下表达非常实用：

- `先分析，不要改代码`
- `先给方案，再实施`
- `只做最小修改`
- `不要引入新依赖`
- `保持现有代码风格`
- `优先复用现有工具函数`
- `改完后运行相关测试`
- `如果有不确定的地方，先说明假设`
- `不要动无关文件`
- `给出根因，不要只给表面修复`

这些话会显著提高可控性。

## 14. 典型实战场景

### 场景 1：快速读懂新仓库

可以这样说：

```text
先阅读这个仓库，然后告诉我：
1. 项目是做什么的
2. 启动命令是什么
3. 主要目录各自负责什么
4. 应该从哪个入口文件开始看
```

### 场景 2：定位线上 bug

```text
请根据报错信息定位问题：用户登录后跳转 500。先找最可能的调用链和根因，再决定是否修改。
```

### 场景 3：补测试

```text
为 src/utils/date.ts 补齐边界测试，沿用当前测试框架和命名风格，改完后运行相关测试。
```

### 场景 4：生成脚本

```text
写一个脚本，批量清理 logs 目录下 30 天前的文件，并附上使用说明。
```

### 场景 5：重构

```text
重构这个模块中的重复分支逻辑，但不要改变对外接口，也不要做跨文件大搬家。
```

## 15. 做代码审查时怎么提问

如果你想让 Codex 做 review，建议说清楚重点。

推荐写法：

```text
请 review 当前改动，重点关注：
1. 潜在 bug
2. 行为回归
3. 错误处理缺失
4. 测试覆盖不足
```

这样的结果通常比“帮我看看有没有问题”更专业。

高质量 review 的特点是：

- 先列发现的问题
- 指出文件和位置
- 说明为什么是风险
- 不会只说“代码整体还不错”

## 16. 做实现任务时怎么提问

实现类任务建议带上这些信息：

- 需求目标
- 不允许改的部分
- 是否允许新增文件
- 是否允许新增依赖
- 是否需要测试
- 是否要兼容旧行为

例如：

```text
在现有 CLI 中新增 `--json` 输出选项。
要求：
- 保持默认文本输出不变
- 不引入新依赖
- 更新相关测试
- 更新 README 中对应说明
```

这会比一句“加个 json 输出”可靠得多。

## 17. 如何判断 Codex 的结果是否靠谱

不要因为它“说得很像”就直接相信。

你应该重点检查：

### 17.1 它是否真的读过相关代码

如果结论完全脱离仓库实际结构，说明上下文可能没建立好。

### 17.2 它是否解释了原因

靠谱的输出通常会说明：

- 为什么改这里
- 根因是什么
- 为什么这个修复不会破坏其他行为

### 17.3 它是否做了验证

例如：

- 运行了哪些测试
- 哪些测试没法跑
- 有没有残留风险

如果没有验证，结论可信度要打折。

## 18. 常见问题与排查

### 18.1 它一直在看文件，不开始修改

可能原因：

- 任务范围太大
- 代码关联复杂
- 你要求它先分析

解决方式：

```text
现在请基于你刚才的分析，直接实施最小修复，并运行相关测试。
```

### 18.2 它修改太多无关代码

说明你的约束不够明确。

下次直接加：

```text
不要改无关文件，不要顺手做格式化或重命名。
```

### 18.3 它没法运行测试

常见原因：

- 依赖没安装
- 环境变量缺失
- 权限受限
- 网络受限

你可以让它先报告阻塞项：

```text
如果测试无法运行，请明确告诉我缺了什么，不要跳过说明。
```

### 18.4 它给出不确定结论

这不一定是坏事。好的 Codex 会在信息不足时说出假设。

你可以要求它：

```text
把确定事实和推测分开写。
```

## 19. 初学者建议的工作习惯

### 19.1 一次只给一个明确任务

不要在一句话里同时塞进：

- 修 bug
- 重构架构
- 加测试
- 写文档
- 做性能优化

最好拆成多轮。

### 19.2 先小后大

先让它完成一个小任务，确认风格和方向对，再继续扩展。

### 19.3 经常要求验证

例如：

```text
改完后运行相关测试，并告诉我结果。
```

### 19.4 对破坏性操作保持谨慎

像下面这些操作要格外小心：

- 大规模删除
- 强制覆盖
- 重置历史
- 改迁移脚本
- 批量重命名公共接口

## 20. 一个建议你直接复用的提问模板

你可以把下面这段保存起来，以后几乎所有任务都能套用：

```text
请帮我处理这个任务。

目标：
[在这里写清楚你要的结果]

上下文：
[补充模块、报错、文件、业务背景]

约束：
- 只做必要修改
- 保持现有代码风格
- 不改无关文件

验证要求：
[例如运行测试 / 不需要运行测试 / 只需静态检查]

交付要求：
[例如修改代码、解释根因、更新文档]
```

## 21. 一个完整示例

假设你在一个前端项目里遇到按钮点击无效的问题。

你可以这样提问：

```text
请帮我定位并修复“提交订单”按钮点击无响应的问题。

要求：
- 先阅读相关组件和事件处理逻辑
- 优先做最小修改
- 不要改 UI 样式
- 改完后运行相关测试；如果没有测试就至少说明你做了什么验证
- 最后告诉我根因和改动点
```

这条指令的优点是：

- 目标明确
- 范围明确
- 修改策略明确
- 验证标准明确
- 输出格式明确

这就是高质量协作。

## 22. 什么时候应该打断它并重新下指令

如果出现以下情况，应该及时纠偏：

- 它开始改与你任务无关的模块
- 它明显误解了需求
- 它准备做大规模重构，而你只想要小修
- 它反复在无关文件中搜索
- 它准备执行高风险操作但理由不充分

你可以直接说：

```text
停止当前方向。只聚焦在 `src/api/order.ts` 和 `src/components/Checkout.tsx`，做最小修复。
```

明确收窄范围通常是最有效的纠偏方式。

## 23. 初学者最容易犯的错误

### 错误 1：任务描述太空

“优化一下”“看一下”“弄一弄”这类描述信息量太低。

### 错误 2：不写约束

不写约束，结果经常会超出你的预期。

### 错误 3：不要求验证

不验证的代码改动，价值会打折。

### 错误 4：把它当成只会聊天的助手

Codex CLI 的价值在于它能实际操作代码库，不只是解释概念。

### 错误 5：对审批不敏感

看到请求就点同意，是不好的习惯。先看命令目的。

---

## 24. 一句话总结使用原则

想让 Codex CLI 用得顺手，记住这 4 点：

1. 任务具体
2. 约束明确
3. 要求验证
4. 对权限敏感

如果你把这四点做到位，Codex CLI 通常会非常高效。

---

## 25. 给初学者的建议路线

如果你刚开始用，建议按这个顺序练习：

1. 让它解释仓库结构
2. 让它解释某个文件
3. 让它做一个很小的改动
4. 让它补一个测试
5. 让它修一个明确报错
6. 再逐渐尝试中等规模重构

不要一上来就让它做高风险、大范围、强依赖上下文的任务。

## 26. 结语

Codex CLI 的正确使用方式，不是“把活全部丢出去”，而是“像带一个能动手的工程师一样协作”。

你给出清晰目标、边界和验证标准，它就更容易产出可用结果。
